# 流程引擎演进说明

## 当前定位

当前系统的审批流程仍然采用后端内置定义驱动，重点是保证现有招生申请、科研报告、外出研修、论文主档四类业务可以稳定流转。

在此基础上，底层存储已经新增一组仿照 Flowable 分层思想的流程表，作为未来演进到可视化流程设计和更强执行能力的兼容层。

## 已落地的存储分层

当前已新增以下几类表：

- 流程模型表：用于承载未来 BPMN.js 编辑后的模型元数据。
- 流程定义与部署表：用于保存发布后的流程定义版本。
- 运行时执行、任务、变量、身份关联表：用于表达当前正在流转的流程状态。
- 历史流程、历史任务、历史活动、历史变量表：用于沉淀审计轨迹和统计分析。

## 与当前实现的关系

现阶段系统仍以业务对象和后端流程定义为主，新增的 Flowable 风格表主要由运行态数据同步生成，作用如下：

- 保持当前功能不回退，现有审批接口继续可用。
- 让流程定义、流程实例、任务实例之间具备稳定的 ID 和关系。
- 统一使用 business_key 连接业务对象与流程实例。
- 为未来流程定义发布、版本切换、BPMN 可视化设计保留结构基础。

## 后续建议路线

建议按以下顺序逐步演进：

1. 新增流程模型管理接口，支持草稿、发布、版本号和启停控制。
2. 将当前内置流程定义逐步抽象成流程定义记录，而不是硬编码常量。
3. 为节点动作增加条件表达式、候选人规则和表单元数据。
4. 接入 BPMN.js，实现流程模型编辑、导入导出和发布。
5. 在运行态中逐步从“业务服务驱动”过渡到“流程定义驱动”。

## 设计原则

- 当前先保证兼容，不一次性替换现有审批服务。
- 运行态与历史态分离，避免审计信息丢失。
- 业务主键统一使用 business_key，避免流程实例和业务记录双主键割裂。
- 流程引擎演进优先解决存储和发布，再考虑复杂 BPMN 语义。
