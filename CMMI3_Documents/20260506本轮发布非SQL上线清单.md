# 20260506 本轮发布非 SQL 上线清单

## 1. 结论

本轮发布不需要执行新的数据库结构迁移，也不需要追加执行 update20260506.sql 或其他新增 SQL。

本轮上线属于“代码发布 + 应用启动后自动修复 + 页面核验”，核心动作如下：

1. 发布后端代码。
2. 发布前端静态资源。
3. 重启后端服务，使启动阶段自动执行既有 schema 校验与工作流运行态修复。
4. 按清单完成功能核验。

## 2. 本轮变更范围

本轮发布实际包含以下内容：

1. 修复驳回后重新提交的报名单在审批中心无法再次审批的问题。
2. 修复系统启动时对历史工作流任务的迁移结果未及时增量回写 PostgreSQL 兼容表的问题。
3. 修复门户申请 V2 中部分附件“URL 已存在但文件名为空时被错误显示为未上传”的页面展示问题。
4. 修复 `management_service_system.py` 的 IDE 告警，不影响生产运行逻辑。

其中第 1、2 项会在新版本服务启动并加载 PostgreSQL 数据后，通过应用逻辑自动完成历史任务修正；不是手工 SQL 补丁。

## 4. 标准上线步骤

建议在低峰窗口按以下顺序执行。

### 4.1 停止应用服务

1. 停止生产后端服务。
2. 如果生产环境由 FastAPI 统一托管前端，建议在切换期间同时暂停外部访问入口，避免用户在旧前端和新后端之间交叉访问。

### 4.2 发布后端代码

将本轮后端代码发布到生产服务器。

本轮后端发布的直接目的：

1. 让门户提交时同步写入当前 workflow task。
2. 让系统启动后能把历史被修正的 workflow task 增量回写到 PostgreSQL 兼容表。

### 4.3 发布前端静态资源

本轮前端页面有实际变更，不能只发后端。

如果生产为统一托管模式，执行前端构建并替换 `frontend/dist`；如果为前后端分离模式，则发布对应静态资源目录。

构建参考命令：

```powershell
Set-Location frontend
npm run build
```

### 4.4 安装依赖

本轮没有新增依赖，通常可跳过。

如生产环境要求每次发版固定重装依赖，可按现网标准执行：

```powershell
.\.venv\Scripts\python.exe -m pip install -r backend\requirements.txt
```

### 4.5 启动新版本后端服务

按现网方式启动后端服务。

本步骤非常关键，因为本轮历史审批任务修复依赖应用启动阶段加载 PostgreSQL 状态后自动执行。

如果生产使用统一托管模式，可参考：

```powershell
.\.venv\Scripts\python.exe -m uvicorn app.main_static:app --app-dir backend --host 0.0.0.0 --port 8000
```

如果生产使用 systemd、Supervisor、Windows Service 或现有脚本，请按现网方式重启。

### 4.6 观察启动日志

启动后重点关注以下异常是否出现：

1. PostgreSQL 连接失败。
2. `ensure_schema()` 失败。
3. 工作流任务加载失败。
4. 启动后出现 workflow task 增量回写异常。

如果日志无异常，即可进入业务核验。

## 5. 上线后必做核验

### 5.1 审批中心核验

至少核验一个“驳回后重提”的真实样本。

本轮建议优先核验报名号 `SH20260008`：

1. 在后台审批中心能看到该报名单的当前待办。
2. 任务状态应为“待处理”。
3. 当前节点应回到 `qualification_review` 对应的资格审核节点。
4. 页面上应可再次执行“通过 / 驳回”动作。

### 5.2 门户填报核验

至少核验以下场景：

1. 学生登录门户后可正常打开申请表。
2. 教育经历章节中，已上传但历史上缺少文件名的附件，页面应显示可点击附件链接，不应再显示为“未上传”。
3. 英语证明、获奖证明、个人简历、其他支撑材料同类场景显示正常。

### 5.3 教育经历保存校验

至少人工验证一次：

1. 清空一个必填教育附件后点击保存。
2. 页面应提示校验失败，不能保存成功。

该项用于确认本轮前端展示修复后，用户不会再因为“已上传但看起来像未上传”而误判校验失效。

## 6. 本轮不需要执行的操作

本轮明确不需要执行以下动作：

1. 不需要新增或追加 update20260506.sql。
2. 不需要执行新的结构迁移 SQL。
3. 不需要执行专门的手工 workflow 补数脚本。
4. 不需要执行门户学生或学生团队的历史补数脚本。

需要注意的是：应用启动后，代码可能会自动把历史错误状态的 workflow task 修正并回写 PostgreSQL 兼容表。这属于预期的数据状态修复，不属于额外人工执行步骤。

## 7. 风险与回滚

### 7.1 风险说明

1. 如果只发后端不发前端，门户附件显示问题不会修复。
2. 如果发版后未重启后端，历史 workflow task 自动修复不会触发。
3. 如果生产存在多实例部署，应确保新版本实例都已切换完成，避免旧实例继续覆盖或读取旧状态。

### 7.2 回滚建议

如本轮发布后发现严重异常，可按以下顺序回滚：

1. 停止当前新版本服务。
2. 回滚后端代码到上一个稳定版本。
3. 回滚前端静态资源到上一个稳定版本。
4. 启动旧版本服务。

通常不需要回滚数据库结构，因为本轮没有结构变更。

如确需回退已被自动修正的个别 workflow 任务状态，应基于上线前数据库备份按记录级别审慎恢复，不建议直接整库回退。

## 8. 最终结论

本轮发布的生产动作可以概括为：

1. 发后端。
2. 发前端。
3. 重启服务。
4. 核验审批中心与门户页面。

除上述动作外，不需要新增 SQL 执行步骤。
