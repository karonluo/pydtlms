## 1. 背景与目标

当前门户已具备博士生在线申请的基础能力，包括：

- 门户登录与会话校验。
- 报名表草稿回显。
- 个人照片、简历、成绩单、学位证、英语证明等附件上传。
- 申请表最终提交。

但现有 [frontend/src/views/portal/StudentPortalApplicationView.vue](frontend/src/views/portal/StudentPortalApplicationView.vue) 与新增的 V2 包装页仍然本质上属于“单页长表单”模式，与目标交互模式不一致。

本计划的目标是将 `applicationv2` 重构为“章节式分步填报页面”，其交互与状态管理遵循新的产品规则，而不再只是对旧页面做视觉包装。

## 2. 对需求的理解

结合本次说明，我对需求的理解如下：

1. `applicationv2` 不再沿用旧版长页面滚动填写模式。
2. 左侧菜单中的每一个菜单项，对应一个独立填写页或独立内容面板。
3. 用户点击左侧菜单时，主内容区要切换到该章节的填写内容，而不是仅滚动到页面中的某个锚点。
4. 点击“下一步”时，本质上等价于切换到左侧菜单中的下一个章节。
5. 点击“保存”时，只保存当前章节内容，不要求一次性填写全部章节。
6. 学生下次登录后，应能恢复到上次保存的草稿状态，继续填写未完成章节。
7. 系统需要识别每个章节的完成状态，只有当所有必填章节均达到完成状态后，才允许执行最终“提交申请表”。
8. 最终提交后，仍需复用现有招生申请提交链路，保持后端业务编号、申请状态、附件数据、草稿数据兼容。

以上理解如无异议，后续开发均以此为准。

## 3. 视觉资源约束

填报页面的背景资源按以下文件为准：

- 顶部背景：`frontend/public/images/top_background.png`
- 底部背景：`frontend/public/images/bottom_backgound.png`

说明：本次用户已明确修正文件名，后续页面样式和资源引用需以这两个名称为准。

## 4. 目标页面形态

`/portal/applicationv2` 应建设为一个“章节式填报工作台”，建议形态如下：

1. 顶部保留门户头部与阶段条，用于展示当前大流程阶段。
2. 左侧为章节导航区，固定展示各个填报章节。
3. 右侧为主编辑区，同一时刻只展示一个章节。
4. 每个章节底部统一提供操作按钮：
   - 保存
   - 上一步
   - 下一步
5. 最后一个章节底部提供：
   - 保存
   - 返回上一步
   - 提交申请表
6. 导航区需可视化展示章节状态，例如：未开始、进行中、已完成。

## 5. 章节拆分建议

建议将 `applicationv2` 拆分为以下章节：

1. 基本信息
2. 报名信息
3. 教育经历
4. 实践经历
5. 英语语言能力
6. 家庭情况
7. 论文发表及获奖经历
8. 个人陈述
9. 提交申请

其中：

- “当前招生计划”可作为页面头部展示信息，不建议再作为单独填写章节。
- “提交申请”建议单独成为最后一章，用于统一展示完成度、声明勾选、最终提交按钮。

## 6. 业务规则定义

### 6.1 保存规则

1. 每个章节支持单独保存。
2. 保存操作写入统一草稿结构 `application_draft`。
3. 即使其他章节未填写，当前章节也应允许保存。
4. 保存成功后更新左侧菜单的章节状态。

### 6.2 切换规则

1. 点击左侧菜单直接切换章节。
2. 点击“下一步”切换至下一个章节。
3. 点击“上一步”切换至上一个章节。
4. 切换章节前若当前章节内容有改动但未保存，需明确策略：
   - 方案 A：自动保存后切换。
   - 方案 B：提示“是否保存后切换”。

建议采用方案 B，更稳妥，便于避免误覆盖。

### 6.3 完成状态规则

每个章节需要定义独立完成条件。建议如下：

1. 基本信息：基础字段齐全，且个人照片已上传。
2. 报名信息：第一志愿研究中心已选。
3. 教育经历：至少一条有效教育经历。
4. 实践经历：非必填，可视为可跳过章节。
5. 英语语言能力：按需求确认为可选或必填，当前建议可跳过。
6. 家庭情况：父亲和母亲信息完整。
7. 论文发表及获奖经历：非必填，可跳过。
8. 个人陈述：个人陈述正文已填写，且声明章节前置数据齐全。
9. 提交申请：声明已勾选，且所有必填章节均完成。

### 6.4 提交规则

1. 当存在未完成的必填章节时，不允许提交。
2. 系统需要明确提示未完成章节名称。
3. 提交成功后，沿用现有 `submitPortalApplication` 接口。
4. 提交后页面应回显提交时间、业务编号、申请状态。

## 7. 技术实现方案

### 7.1 前端结构调整

建议新增以下结构：

1. `frontend/src/views/portal/StudentPortalApplicationV2View.vue`
   - 作为新的章节式填报主页面。
2. `frontend/src/views/portal/application-v2/`
   - 存放分章节组件。
3. 章节组件建议拆分为：
   - `ApplicationV2BasicSection.vue`
   - `ApplicationV2ApplySection.vue`
   - `ApplicationV2EducationSection.vue`
   - `ApplicationV2PracticeSection.vue`
   - `ApplicationV2EnglishSection.vue`
   - `ApplicationV2FamilySection.vue`
   - `ApplicationV2AchievementSection.vue`
   - `ApplicationV2StatementSection.vue`
   - `ApplicationV2SubmitSection.vue`

### 7.2 状态管理建议

建议将 `applicationv2` 的共享逻辑抽到单独 composable：

1. `frontend/src/composables/usePortalApplicationForm.ts`
   - 统一管理表单数据。
   - 统一管理章节状态。
   - 统一管理保存、切换、提交。
2. `frontend/src/composables/usePortalApplicationSections.ts`
   - 定义章节元数据。
   - 定义完成条件。
   - 定义前进后退逻辑。

### 7.3 接口策略

优先复用现有接口，不新增无必要后端接口：

1. `getPortalProfile`
2. `listPortalPlans`
3. `listPortalTeams`
4. `getPortalProfileOptions`
5. `uploadPortalAttachment`
6. `submitPortalApplication`

如需支持“保存当前章节”与“记录章节完成状态”，建议先在前端基于现有草稿结构实现；若现有后端无法满足更细粒度保存，再补充草稿保存接口。

## 8. 推荐迭代顺序

### 第一阶段：结构重构

目标：把 `applicationv2` 从长页面改成章节切换模式。

任务：

2. 搭建左侧导航 + 右侧章节区。
3. 建立章节元数据与切换机制。
4. 去除锚点滚动逻辑。

### 第二阶段：数据与保存

目标：实现“当前章节保存”和“重新登录继续填写”。

任务：

1. 抽离共享表单状态。
2. 实现章节级保存。
3. 完成草稿回填。
4. 记录章节完成状态。

### 第三阶段：提交闭环

目标：实现“所有必填章节完成后才可提交”。

任务：

1. 定义章节完成校验规则。
2. 构建章节完成度展示。
3. 在提交页集中展示未完成项。
4. 复用现有提交接口完成最终提交。

### 第四阶段：视觉收口

目标：与目标图进一步贴合。

任务：

1. 接入顶部/底部背景图。
2. 调整左侧菜单视觉。
3. 调整表单宽度、列数、按钮区样式。
4. 完成移动端适配与交互细节优化。

## 9. 风险与注意事项

1. 不能只做页面切换效果，而忽略草稿保存与章节完成状态，否则会偏离本次核心需求。
2. 不能破坏现有提交接口的数据结构兼容性。
3. `application_status` 属于流程托管字段，不能在前端章节切换或草稿保存中被误修改。
4. 附件上传状态需要和当前章节保存协同处理，避免“附件刚上传但章节未保存”导致回填异常。
5. 若顶部/底部背景资源文件名已被更正，样式引用需同步修正，避免旧文件名残留。
