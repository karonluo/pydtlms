# 生产环境门户学生数据修复上线步骤

## 1. 适用范围

本文用于本次“门户学生注册数据正式落表修复”的生产环境迭代上线。

本次修复内容包括：

1. 门户学生注册、草稿保存、状态变更时，同步写入正式列式表 `dtlms_portal_students`。
2. 门户学生扩展资料同步写入 `dtlms_portal_student_profiles`。
3. 保留 `dtlms_runtime_portal_students` 作为运行态镜像表。
4. 新增历史补数脚本，可将既有 runtime JSONB 数据回填到正式列式表。
5. 新增幂等 SQL 迁移，应用重启时可自动回填历史数据。

## 2. 涉及文件

本次上线涉及的核心文件如下：

1. `backend/app/services/management_service.py`
2. `backend/app/services/postgres_state_store.py`
3. `backend/sql/027_portal_student_runtime_backfill.sql`
4. `backend/scripts/backfill_portal_students_to_relational.py`

## 3. 上线前准备

### 3.1 备份数据库

上线前必须先对生产数据库做完整备份。

建议使用 PostgreSQL 官方工具执行：

```powershell
pg_dump -h <生产库地址> -p <端口> -U <用户名> -d <数据库名> -F c -b -v -f dtlms_prod_before_portal_student_fix.backup
```

如需只做门户相关逻辑的应急快照，至少备份以下表：

1. `dtlms_runtime_portal_students`
2. `dtlms_portal_students`
3. `dtlms_portal_student_profiles`
4. `dtlms_runtime_operation_logs`
5. `dtlms_operation_logs`

### 3.2 确认应用配置

确认生产环境 `backend/.env` 已指向生产 PostgreSQL。

重点核对：

1. `POSTGRES_HOST`
2. `POSTGRES_PORT`
3. `POSTGRES_USER`
4. `POSTGRES_PASSWORD`
5. `POSTGRES_DB`

### 3.3 确认代码版本

确认生产待发布版本已包含以下改动：

1. 门户学生双写正式表逻辑
2. `027_portal_student_runtime_backfill.sql`
3. `backfill_portal_students_to_relational.py`

## 4. 标准上线步骤

### 4.1 停止应用服务

按现网部署方式停止后端服务。

如果前后端同机部署，建议同时停止前端访问入口，避免上线中产生新的注册写入。

### 4.2 发布代码

将本次后端代码发布到生产服务器。

如果生产前端需要一起发版，则同步发布前端静态资源；如果本次只涉及后端，可只发布后端。

### 4.3 安装依赖

如生产环境依赖已固化且本次无新增依赖，可跳过。

如需安装依赖，执行：

```powershell
.\.venv\Scripts\python.exe -m pip install -r backend\requirements.txt
```

### 4.4 执行历史数据只读预检查

先执行 dry-run，确认待补数记录量：

```powershell
.\.venv\Scripts\python.exe backend\scripts\backfill_portal_students_to_relational.py --dry-run --summary
```

预期输出应包含：

1. `loaded_from_runtime`
2. `processed`
3. `skipped`
4. `invalid_students`

要求：

1. `loaded_from_runtime` 应大于等于当前 runtime 门户学生记录数。
2. `skipped` 如不为 0，需先查看 `invalid_students` 中缺失字段情况。

### 4.5 启动后端服务并执行自动迁移

启动后端服务后，系统会自动执行 `ensure_schema()`，并自动执行迁移 SQL，包括：

1. `027_portal_student_runtime_backfill.sql`

如生产使用 Windows PowerShell 启动，可参考：

```powershell
.\.venv\Scripts\python.exe -m uvicorn app.main:app --app-dir backend --host 0.0.0.0 --port 8000
```

如果生产使用现有脚本启动，可按现网方式启动。

### 4.6 执行一次人工补数脚本

虽然应用启动时已经会自动执行 SQL 迁移，但生产建议额外执行一次 Python 补数脚本，确保 runtime 数据和正式表完全对齐。

执行命令：

```powershell
.\.venv\Scripts\python.exe backend\scripts\backfill_portal_students_to_relational.py --summary
```

如需单个学生补数，可执行：

```powershell
.\.venv\Scripts\python.exe backend\scripts\backfill_portal_students_to_relational.py --student-id 123 --summary
```

## 5. 上线后核查步骤

### 5.1 核查正式表记录数

执行以下 SQL：

```sql
SELECT COUNT(*) AS runtime_count FROM dtlms_runtime_portal_students;

SELECT COUNT(*) AS relational_count FROM dtlms_portal_students;
```

要求：

1. `relational_count` 不应明显小于 `runtime_count`。

### 5.2 核查最近注册用户是否已落正式表

```sql
SELECT id, full_name, phone_number, email, id_number, account_status, created_at, updated_at
FROM dtlms_portal_students
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

### 5.3 核查扩展资料表

```sql
SELECT portal_student_id, gender, birth_date, ethnic_group, native_place, political_status, updated_at
FROM dtlms_portal_student_profiles
ORDER BY updated_at DESC NULLS LAST, portal_student_id DESC
LIMIT 20;
```

### 5.4 核查后台管理端“注册学生”页面

人工核对后台管理端：

1. 刚注册学生可查询到
2. 账号状态显示正常
3. 招生计划、导师团队、报名状态显示正常

### 5.5 核查新增注册链路

建议用一名测试考生完成以下动作：

1. 门户注册
2. 门户登录
3. 保存申请草稿
4. 后台管理端查看注册学生

然后执行 SQL 验证：

```sql
SELECT id, full_name, phone_number, email, selected_plan_id, selected_team_name, selected_advisor_name, submitted_at
FROM dtlms_portal_students
WHERE phone_number = '<测试手机号>';

SELECT portal_student_id, gender, birth_date, political_status, mailing_address
FROM dtlms_portal_student_profiles
WHERE portal_student_id = <上一步查出的 id>;
```

## 6. 异常处理

### 6.1 dry-run 出现 skipped > 0

处理方式：

1. 查看脚本输出中的 `invalid_students`
2. 确认缺失字段是否为历史脏数据
3. 如仅个别记录异常，可按 `--student-id` 单独处理

### 6.2 自动迁移已执行但正式表仍缺数据

处理方式：

1. 手工执行 Python 补数脚本
2. 再次核查 `dtlms_portal_students`
3. 查看后端启动日志和脚本输出

### 6.3 页面仍显示不全

处理方式：

1. 先确认正式表是否已存在对应数据
2. 再确认后台接口返回字段是否覆盖页面展示字段
3. 最后确认前端页面列定义是否已刷新为最新版本

## 7. 回滚方案

如上线后发现严重异常，可按以下顺序回滚：

1. 停止当前应用服务
2. 回滚后端代码到上一个稳定版本
3. 使用上线前备份恢复数据库
4. 重启应用服务

如不希望整库回滚，也可只恢复门户相关表：

1. `dtlms_runtime_portal_students`
2. `dtlms_portal_students`
3. `dtlms_portal_student_profiles`

## 8. 后续整改建议

本次已先修复门户学生数据正式落表问题，但仓库内仍存在较多 `dtlms_runtime_*` JSONB 运行态表。

后续建议按模块分批整改：

1. 学生模块
2. 招生模块
3. 培养模块
4. 学位模块
5. 审批流程运行态模块

整改原则：

1. 默认使用关系型列式建模
2. 主业务数据不得再以 JSONB 作为唯一事实来源
3. JSONB 仅用于明确批准的扩展字段、快照字段或流程变量
