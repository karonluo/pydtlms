# 数据库列式化改造清单

## 1. 目标

本清单用于指导博士生生命周期管理系统逐步从“runtime JSONB 主存储”过渡为“关系型列式主存储”。

改造原则如下：

1. 主业务数据必须以关系型列式表为唯一事实来源。
2. JSONB 仅允许用于明确批准的扩展字段、快照字段、日志快照或流程变量。
3. 所有数据库结构调整必须同时提供初始化 SQL、迁移 SQL、补数脚本和回滚策略。
4. 所有生产更新必须优先遵循“可回滚、可验证、默认安全”的原则。

### 第二阶段：历史数据补齐

目标：

1. 对每个 runtime JSONB 表，提供一套历史数据回填脚本。
2. 确保正式表行数、关键字段覆盖率、核心索引和外键关系满足上线要求。

输出物要求：

1. 幂等 SQL 迁移
2. Python 补数脚本
3. dry-run 预检查
4. 核查 SQL

### 第三阶段：读链路切换

目标：

1. 所有业务查询、列表、详情、统计接口切换为正式列式表。
2. runtime JSONB 表仅保留镜像/兼容作用，不再作为页面主读取来源。

输出物要求：

1. 查询 SQL 调整
2. API 返回模型校验
3. 前端字段联调验证

### 第四阶段：写链路收口

目标：

1. 新增、编辑、删除操作统一直接维护正式关系表。
2. runtime JSONB 仅在确有必要时作为镜像快照保留。

输出物要求：

1. 服务层重构
2. 单元测试/回归脚本
3. 灰度核查方案

### 第五阶段：移除 JSONB 主存储依赖

目标：

1. 停止把 runtime JSONB 表作为业务主数据承载层。
2. 保留少量经批准的 JSONB 场景，例如：日志快照、流程变量、扩展字段。

## 5. 每张表的统一改造模板

每个 runtime JSONB 表改造时，必须按以下顺序执行：

1. 设计正式列式主表与关系表。
2. 补初始化 SQL。
3. 补迁移 SQL。
4. 补 Python 补数脚本。
5. 改写写入链路，先双写。
6. 改写读取链路，优先正式表。
7. 做 dry-run 检查。
8. 做生产核查 SQL。
9. 编写回滚方案。

## 6. 生产更新原则

所有生产更新必须满足：

1. 默认只读，不自动破坏数据。
2. 必须先备份，再执行迁移或补数。
3. 任何“执行写操作”的脚本都必须要求显式参数确认。
4. 任何补数脚本都必须支持 dry-run。
5. 任何迁移脚本都必须幂等。
6. 任何上线动作都必须附带核查 SQL 和回滚方案。
