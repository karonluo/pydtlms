# 博士生生命周期管理系统 Linux 部署手册

## 1. 适用范围

本文档用于指导博士生生命周期管理系统（DTLMS）在 Linux 服务器上的部署、初始化、启动、升级与排障。

本文默认采用以下部署模式：

- 操作系统：Ubuntu 22.04 / 24.04 或同类 Linux 发行版
- 部署方式：前端静态文件构建后，由 FastAPI 统一托管
- 进程管理：systemd
- 反向代理：Nginx
- 数据库：PostgreSQL
- 缓存/会话：Redis 单机或 Redis Sentinel

该模式的优点是结构简单、运维路径清晰，适合作为生产环境首选方案。

## 2. 部署架构

推荐部署拓扑如下：

1. Nginx 对外提供 80/443 访问入口。
2. Nginx 将请求转发到本机 FastAPI 服务。
3. FastAPI 使用 `app.main_static:app` 启动，同时提供：
   - `/api/v1/*` 后端接口
   - `/health` 健康检查
   - `/` 管理端与学生门户前端页面
   - `/portal-attachments/*` 门户附件静态访问
   - `/portal-brochures/*` 招生简章图片访问
4. FastAPI 连接 PostgreSQL 与 Redis（单机或 Sentinel）。

不建议生产环境继续使用 Vite 开发服务器或前端开发代理模式。

## 3. 目录约定

以下为建议部署目录，可根据实际情况调整：

```text
/opt/pydtlms/
├─ app/                    # Git 仓库代码
├─ .venv/                  # Python 虚拟环境
├─ logs/                   # 运行日志
└─ backups/                # 备份目录
```

本文后续命令均默认项目部署路径为：

```bash
/opt/pydtlms/app
```

## 4. 环境要求

### 4.1 基础软件

建议安装以下基础组件：

- Git
- Python 3.12 或与当前项目兼容的 Python 3 版本
- Python venv
- Node.js 20 LTS
- npm
- Nginx
- PostgreSQL 客户端工具

Ubuntu 示例命令如下：

```bash
sudo apt update
sudo apt install -y git nginx curl build-essential \
  python3 python3-venv python3-pip \
  postgresql-client
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install -y nodejs
```

可通过以下命令检查版本：

```bash
python3 --version
node -v
npm -v
nginx -v
```

### 4.2 外部服务要求

部署前需准备好以下外部服务连接信息：

1. PostgreSQL 数据库地址、端口、库名、用户名、密码。
2. Redis 连接参数。
  - 单机模式：主机、端口、库、认证密码。
  - 哨兵模式：Sentinel 节点列表、主节点名称、认证密码。
3. SMTP 邮件服务器参数。
4. 对外访问域名或服务器 IP。

## 5. 获取代码与创建运行目录

### 5.1 创建部署目录

```bash
sudo mkdir -p /opt/pydtlms
sudo chown -R $USER:$USER /opt/pydtlms
cd /opt/pydtlms
```

### 5.2 拉取代码

```bash
git clone <你的仓库地址> app
cd /opt/pydtlms/app
```

如果服务器上通过压缩包部署，也可直接上传代码到该目录。

## 6. 安装项目依赖

### 6.1 安装 Python 依赖

项目当前约定 Python 虚拟环境位于仓库根目录 `.venv`。

```bash
cd /opt/pydtlms/app
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -r backend/requirements.txt
```

### 6.2 安装前端依赖并构建

```bash
cd /opt/pydtlms/app/frontend
npm install
npm run build
```

构建成功后将生成：

```text
frontend/dist/
```

该目录将由 `backend/app/main_static.py` 统一挂载。

## 7. 后端配置

### 7.1 配置文件位置

后端运行时读取以下文件：

- `backend/.env`
- `backend/.env.local`

通常建议：

1. `backend/.env` 保存基础配置
2. `backend/.env.local` 保存生产环境覆盖项或敏感参数

### 7.2 初始化配置文件

```bash
cd /opt/pydtlms/app
cp backend/.env.example backend/.env
```

### 7.3 必填配置项

至少需要根据生产环境修改以下变量：

```env
APP_NAME=DTLMS Platform
ENVIRONMENT=production
API_V1_PREFIX=/api/v1

POSTGRES_HOST=127.0.0.1
POSTGRES_PORT=5432
POSTGRES_USER=postgres
POSTGRES_PASSWORD=请替换为实际密码
POSTGRES_DB=db_dtlms
POSTGRES_SCHEMA=public

JWT_SECRET_KEY=请替换为高强度随机密钥
JWT_ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=30
REFRESH_TOKEN_EXPIRE_MINUTES=10080

REDIS_MODE=single
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_DB=0
REDIS_HOST_LIST=
REDIS_PASSWORD=请替换为实际密码
REDIS_SENTINEL_NAME=
REDIS_KEY_PREFIX=CTDTLMS_

ALLOWED_ORIGINS=https://your-domain.example.com

DEFAULT_ADMIN_USERNAME=admin
DEFAULT_ADMIN_PASSWORD=请替换为安全密码
```

### 7.4 邮件服务配置

当前系统包含门户注册、找回密码、招生通知等邮件能力，生产环境建议显式配置 SMTP：

```env
SMTP_ENABLED=true
SMTP_HOST=smtp.example.com
SMTP_PORT=465
SMTP_USERNAME=no-reply@example.com
SMTP_PASSWORD=请替换为实际密码
SMTP_FROM_EMAIL=no-reply@example.com
SMTP_FROM_NAME=博士生生命周期管理系统
SMTP_USE_TLS=false
SMTP_USE_SSL=true
SMTP_TIMEOUT_SECONDS=120
```

### 7.5 Redis 配置方式

系统当前支持两种 Redis 配置模式，由 `REDIS_MODE` 控制。

#### 7.5.1 单机 Redis

适用于单节点 Redis、云 Redis 单实例、或由代理统一暴露出的单入口 Redis。

```env
REDIS_MODE=single
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_DB=0
REDIS_PASSWORD=请替换为实际密码
REDIS_HOST_LIST=
REDIS_SENTINEL_NAME=
REDIS_KEY_PREFIX=CTDTLMS_
```

#### 7.5.2 Redis Sentinel

适用于 Redis 高可用场景。`REDIS_HOST_LIST` 填写 Sentinel 节点列表，`REDIS_SENTINEL_NAME` 填写主节点名称。

```env
REDIS_MODE=sentinel
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_DB=0
REDIS_HOST_LIST=127.0.0.1:26379,127.0.0.1:26380,127.0.0.1:26381
REDIS_PASSWORD=请替换为实际密码
REDIS_SENTINEL_NAME=mymaster
REDIS_KEY_PREFIX=CTDTLMS_
```

### 7.6 重要说明

1. 不要直接使用仓库中的默认口令、默认数据库地址和默认 SMTP 参数。
2. `JWT_SECRET_KEY` 必须替换为高强度随机字符串。
3. `ALLOWED_ORIGINS` 需要设置为生产域名，多个来源用英文逗号分隔。
4. 生产环境不需要开启 `FRONTEND_DEV_PROXY_ENABLED`。

## 8. 数据库初始化

### 8.1 执行初始化脚本

项目已提供一键初始化脚本：

```bash
cd /opt/pydtlms/app
source .venv/bin/activate
python backend/scripts/init_postgres.py
```

该脚本会根据当前运行态将结构和基础数据同步到 PostgreSQL。

如只需查看摘要，可执行：

```bash
python backend/scripts/init_postgres.py --summary
```

### 8.2 初始化内容

初始化过程会涉及以下内容：

1. PostgreSQL 结构初始化
2. 研究中心、学生、招生、培养、学位等业务表准备
3. RBAC 基础数据准备
4. 字典数据准备
5. 运行态数据同步

### 8.3 首次部署建议

首次上线前建议至少完成以下检查：

1. 管理员账号是否已创建。
2. PostgreSQL 是否能正常查询基础表。
3. Redis 是否可用。
  - 单机模式：确认 `REDIS_HOST:REDIS_PORT` 可达。
  - 哨兵模式：确认 Sentinel 能发现主节点。
4. `/health` 是否返回 `ok`。

## 9. 启动方式

### 9.1 生产模式启动命令

生产环境建议直接使用以下命令启动：

```bash
cd /opt/pydtlms/app
source .venv/bin/activate
uvicorn app.main_static:app --app-dir backend --host 127.0.0.1 --port 8000
```

说明如下：

1. `app.main_static:app` 用于同时提供 API 与已构建前端页面。
2. `--app-dir backend` 表示 Python 模块根目录为 `backend`。
3. `--host 127.0.0.1` 表示仅本机监听，由 Nginx 对外代理。
4. `--port 8000` 为后端监听端口，可按需调整。

### 9.2 为什么不建议生产使用开发模式

以下模式仅适合本地开发，不适合作为正式部署方式：

- `uvicorn app.main:app --reload`
- `npm run dev`
- FastAPI 前端开发代理模式

原因包括：

1. 进程不稳定
2. 性能不适合生产
3. 依赖开发工具链
4. 运维复杂度更高

## 10. 使用 systemd 托管服务

### 10.1 创建日志目录

```bash
mkdir -p /opt/pydtlms/logs
```

### 10.2 创建服务文件

创建文件：

```text
/etc/systemd/system/pydtlms.service
```

建议内容如下：

```ini
[Unit]
Description=DTLMS FastAPI Service
After=network.target

[Service]
Type=simple
User=www-data
Group=www-data
WorkingDirectory=/opt/pydtlms/app
Environment=PYTHONUNBUFFERED=1
ExecStart=/opt/pydtlms/app/.venv/bin/python -m uvicorn app.main_static:app --app-dir backend --host 127.0.0.1 --port 8000
Restart=always
RestartSec=5
StandardOutput=append:/opt/pydtlms/logs/pydtlms.out.log
StandardError=append:/opt/pydtlms/logs/pydtlms.err.log

[Install]
WantedBy=multi-user.target
```

如果实际运行用户不是 `www-data`，请替换为对应账户。

### 10.3 启用服务

```bash
sudo systemctl daemon-reload
sudo systemctl enable pydtlms
sudo systemctl start pydtlms
sudo systemctl status pydtlms
```

### 10.4 常用运维命令

```bash
sudo systemctl restart pydtlms
sudo systemctl stop pydtlms
sudo systemctl status pydtlms
journalctl -u pydtlms -f
```

## 11. 使用 Nginx 提供对外访问

### 11.1 创建站点配置

创建文件：

```text
/etc/nginx/sites-available/pydtlms.conf
```

参考配置如下：

```nginx
server {
    listen 80;
    server_name your-domain.example.com;

    client_max_body_size 100m;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 300;
    }
}
```

### 11.2 启用站点

```bash
sudo ln -s /etc/nginx/sites-available/pydtlms.conf /etc/nginx/sites-enabled/pydtlms.conf
sudo nginx -t
sudo systemctl reload nginx
```

### 11.3 HTTPS 建议

生产环境建议使用 HTTPS，可结合 Certbot 部署：

```bash
sudo apt install -y certbot python3-certbot-nginx
sudo certbot --nginx -d your-domain.example.com
```

## 12. 附件与静态资源注意事项

系统静态资源涉及以下目录：

```text
frontend/dist/
frontend/public/portal-attachments/
frontend/public/portal-brochures/
```

说明如下：

1. `frontend/dist/` 为前端构建产物。
2. `frontend/public/portal-attachments/` 为门户上传附件的公开访问目录。
3. `frontend/public/portal-brochures/` 为招生简章图片目录。

部署时需要注意：

1. 升级代码前先备份 `frontend/public/portal-attachments/`。
2. 若部署方式会覆盖整个项目目录，应单独保留附件目录。
3. 执行 `npm run build` 不会自动删除 `frontend/public` 原始文件，但发布脚本若采用覆盖式同步，仍可能误删附件。

部署完成后建议按以下顺序验证：

### 13.1 服务级检查

```bash
curl http://127.0.0.1:8000/health
```

期望返回：

```json
{"status":"ok","service":"DTLMS Platform","environment":"production"}
```

### 13.2 页面级检查

访问以下地址：

1. 管理端首页
2. 学生门户登录页 `/portal`
3. 学生门户申请页 `/portal/application`
4. OpenAPI 文档 `/docs`（如生产环境允许暴露）

### 13.3 登录检查

建议验证以下场景：

1. 管理员登录
2. 学生门户注册
3. 学生门户登录
4. 学生找回密码
5. 报名表保存与提交

## 14. 升级流程

建议采用如下升级流程：

1. 备份数据库。
2. 备份 `frontend/public/portal-attachments/`。
3. 拉取最新代码。
4. 更新 Python 依赖。
5. 更新前端依赖并重新构建。
6. 如有结构变化，执行数据库初始化或迁移脚本。
7. 重启 systemd 服务。
8. 执行健康检查与核心功能验证。

参考命令：

```bash
cd /opt/pydtlms/app
git pull
source .venv/bin/activate
pip install -r backend/requirements.txt
cd frontend
npm install
npm run build
cd ..
python backend/scripts/init_postgres.py --summary
sudo systemctl restart pydtlms
curl http://127.0.0.1:8000/health
```

## 15. 常见问题排查

### 15.1 后端启动失败

检查项：

1. `backend/.env` 是否存在。
2. PostgreSQL 地址、库名、密码是否正确。
3. Redis 地址是否可达，且与 `REDIS_MODE` 配置一致。
4. Python 依赖是否完整安装。
5. systemd 服务工作目录是否正确。

### 15.2 页面能打开但接口报错

检查项：

1. `ALLOWED_ORIGINS` 是否包含生产域名。
2. Nginx 是否正确转发到 8000 端口。
3. FastAPI 是否使用 `app.main_static:app` 启动。
4. 浏览器控制台与 Nginx 日志是否存在 502/504。

### 15.3 登录失败或会话异常

检查项：

1. `REDIS_MODE` 是否与实际部署方式一致。
2. 单机模式下，`REDIS_HOST`、`REDIS_PORT`、`REDIS_DB` 是否正确。
3. 哨兵模式下，Redis Sentinel 是否能发现主节点。
4. 哨兵模式下，`REDIS_SENTINEL_NAME` 是否与实际主名称一致。
5. `REDIS_PASSWORD` 是否正确。
6. 服务器是否存在防火墙拦截。

### 15.4 邮件发送失败

检查项：

1. SMTP 用户名和密码是否正确。
2. SMTP SSL/TLS 模式是否与实际服务一致。
3. 服务器对外网络是否允许访问邮件服务器端口。
4. 发件地址是否具备发信权限。

### 15.5 附件无法访问

检查项：

1. `frontend/public/portal-attachments/` 是否存在。
2. 运行用户是否有读取权限。
3. Nginx 是否正确代理到 FastAPI。
4. 上传后的相对路径是否仍指向当前部署目录。

## 16. 生产部署建议

1. 使用独立 Linux 用户运行服务，避免使用 root 直接启动。
2. 生产环境务必替换所有默认密码、JWT 密钥和 SMTP 默认参数。
3. 建议通过 HTTPS 暴露系统。
4. 建议定期备份 PostgreSQL 数据与门户附件目录。
5. 建议将 `backend/.env` 纳入运维配置管理，不要提交到 Git。
6. 建议使用 systemd + Nginx 作为标准部署方式，避免手工后台启动。

## 17. 建议的上线前检查清单

上线前建议逐项确认：

1. 前端构建成功。
2. FastAPI 健康检查通过。
3. PostgreSQL 连接正常。
4. Redis 会话正常。
5. 管理员登录正常。
6. 学生门户注册、登录、找回密码正常。
7. 报名表保存、提交、附件访问正常。
8. Nginx 代理与 HTTPS 正常。
9. 日志目录可写。
10. 备份策略已建立。
