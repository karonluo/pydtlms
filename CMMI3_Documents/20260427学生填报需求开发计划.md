## 1. 文档目的

本文用于分析 [需求/原始需求资料/2026.04.27 需求会议单.txt](需求/原始需求资料/2026.04.27%20需求会议单.txt) 中新增的学生门户填报需求，并形成后续逐步开发的实施计划、影响范围、优先级与验收口径。

本文聚焦“学生门户报名填报”和“学生门户登录”两个范围，不包含招生工作台、培养学位流程或后台审批流的扩展设计。

## 2. 当前现状

结合现有代码，学生门户相关能力已经具备以下基础：

- 登录/注册/找回密码页面已存在，入口位于 [frontend/src/views/portal/StudentPortalAuthView.vue](frontend/src/views/portal/StudentPortalAuthView.vue)。
- 学生报名 V2 表单已拆分为章节式页面，主入口位于 [frontend/src/views/portal/applicationv2/PortalApplicationV2Form.vue](frontend/src/views/portal/applicationv2/PortalApplicationV2Form.vue)。
- 各章节组件已拆出，主要包括基本信息、报名信息、教育经历、实践经历、英语能力、家庭成员、成果经历、个人陈述等分段组件，目录位于 [frontend/src/views/portal/applicationv2/sections](frontend/src/views/portal/applicationv2/sections)。
- 后端门户接口集中在 [backend/app/api/v1/portal.py](backend/app/api/v1/portal.py)。
- 门户数据结构定义集中在 [backend/app/schemas/portal.py](backend/app/schemas/portal.py)。
- 门户报名草稿、正式提交和子表持久化已落到 PostgreSQL 关系表与兼容镜像中，主存储逻辑在 [backend/app/services/postgres_state_store.py](backend/app/services/postgres_state_store.py) 与 [backend/app/services/management_service.py](backend/app/services/management_service.py)。

这意味着，本次不是从零开发，而是在既有门户报名链路上做“字段、交互和校验规则升级”。

## 3. 需求分析结论

### 3.1 基本信息

新增或明确以下要求：

1. 身份证照片需支持上传正反面拼接图。
2. 文件格式要求为 JPG。
3. 需要提供明确示例，表达为“一半个人照，一半身份证照片”。

分析：

- 这部分主要影响基本信息章节的上传说明、附件校验和展示文案。
- 当前后端附件分类未单独定义“身份证拼接图”，需要评估是否复用现有头像/附件分类，或新增专用附件分类。

### 3.2 报名信息

新增或调整以下要求：

1. “了解项目方式”位置移动到研究中心志愿下方。
2. 研究中心志愿直接展示两个志愿，第二志愿可选。
3. 志愿标题显示为“第一志愿”“第二志愿”，使用中文数字。
4. 标题文案调整为“研究领域选择”，但下方实际选择项仍是研究中心。
5. 明确标注“第一志愿必填，第二志愿选填”。

分析：

- 前端当前 `preferences` 是数组结构，已天然支持两个志愿，但现有文案、布局和强约束仍需调整。
- 这部分主要是前端章节组件和前端校验规则调整，后端数据结构可基本复用。

状态：已完成。

### 3.3 教育经历

新增或调整以下要求：

1. 教育经历类型改为：硕士毕业、硕士在读、本科毕业、本科在读、高中毕业。
2. 至少填写两条，建议顺序为“硕士/本科/高中”。
3. 至多填写四条。
4. 高中阶段隐藏成绩、学位、专业控件。
5. 初始进入页面时，默认生成“教育经历1：高中毕业”和“教育经历2：空白”。
6. 教育经历2不可留空，且必须完整填写；其教育阶段只能为“本科在读”或“本科毕业”。
7. 后续新增教育经历时，应从本科在读位置继续补加，而不是无序追加。
8. 保存和提交时增加联动校验：
   - 教育经历2为空、学校未填，或教育阶段不是“本科在读/本科毕业”时，不允许保存和提交。
   - 如果填写了本科毕业，则必须填写硕士在读或硕士毕业，否则不允许保存和提交。
   - 如果填写了本科在读，则不能增加硕士在读或硕士毕业。

分析：

- 这是本轮变动最大的部分，涉及前端初始数据、表单显隐、前端校验、后端 schema 校验、保存/提交校验和草稿回填逻辑。
- 当前教育经历结构已关系化存储，但枚举值和规则不符合新需求，需要统一升级前后端。

状态：已完成。

### 3.4 实践经历

新增或调整以下要求：

1. 最多 2 条。
2. 默认保留 1 条空白实践经历。
3. 存储时空白记录不落库。
4. 当填写了开始年月或结束年月时，除职责外其余字段均必填。
5. 如有更多内容，引导用户通过附件上传补充。

分析：

- 当前实践经历已完成默认 1 条空白占位、最多 2 条、保存/提交统一校验，以及“更多内容请在个人陈述与附件补充上传”的前端引导。
- 后端 schema 已统一过滤空白占位记录，避免草稿和正式提交把前端保留行直接持久化。

状态：已完成。

### 3.5 英语能力

新增或调整以下要求：

1. 名称从“英语语言能力”统一为“英语能力”。
2. 该章节必填。
3. 默认保留一套空白记录。
4. 去掉 CET-4。
5. 英语证明附件必填。

分析：

- 当前已完成“英语语言能力”统一更名为“英语能力”，并默认保留 1 条空白记录。
- 前端已移除 CET-4，提交阶段会校验至少 1 条英语能力，且英语证明附件必填。
- 后端 schema 已同步增加提交兜底校验，避免绕过前端直接提交空英语记录或缺少附件的数据。

状态：已完成。

### 3.6 家庭成员

新增或调整以下要求：

1. 父母至少一方必填。

分析：

- 当前 V2 页面默认保留父亲和母亲两条记录，提交校验已改为“父母至少一方必填”。
- 前端章节文案、完成状态判断和提交阻断提示已同步调整；后端 schema 已补同口径兜底校验。

状态：已完成。

### 3.7 登录页面

新增或调整以下要求：

1. 登录方式改为“账号登录 / 验证码登录（通过邮件）”二选一。

分析：

- 当前登录仅支持账号密码登录；邮箱验证码目前只用于注册校验，不支持登录。
- 这是一个独立功能块，涉及前端模式切换、后端发送验证码接口复用或扩展、验证码校验、登录 token 签发链路调整。

### 3.8 成果经历

新增或调整以下要求：

1. 原“获奖经历”改为“成果经历”。
2. 最多填写 4 条。
3. 当类型为“论文发表”时，字段为：日期（年/月）、论文名称、作者序位、期刊名称、描述。
4. 当类型为“获奖经历”时，字段为：日期（年/月）、奖项名称、获奖名次、获奖证明上传、描述。

分析：

- 当前成果经历数据结构字段与新需求不完全一致。
- 后端 schema、前端表单、子表持久化都要扩展字段。
- 这是本轮除教育经历外第二个结构性改动点。

### 3.9 个人陈述与附件

新增或调整以下要求：

1. 个人陈述内容需按 Excel 中 Subject QA 口径调整。
2. 个人简历附件必传。
3. 其他支撑材料改为单文件上传，明确提示建议使用 zip 压缩，且为选填。

当前状态：已完成。

实现结果：

- 门户 V2 个人陈述章节已改为“个人成长经历、为何申报本项目或本专业、未来职业发展规划”三段主题式填写，并在界面明确提示总字数 800-1200、中英文皆可。
- 个人简历附件已改为提交必传，前后端均已加入校验。
- 其他支撑材料已改为单文件上传，前端明确提示“建议使用 zip 压缩、选填”，后端已新增 supporting_material 附件类别。
- PostgreSQL 个人陈述关系表已扩展为结构化字段；迁移脚本为 [backend/sql/055_portal_personal_statement_v2.sql](backend/sql/055_portal_personal_statement_v2.sql)，并已幂等合并进生产总补丁 [backend/sql/900_safe_production_update_columnar.sql](backend/sql/900_safe_production_update_columnar.sql)。
- 已完成 portal API 测试与 state store 单测回归，前端构建通过。

## 4. 开发范围划分

本次需求建议拆为 5 个开发主题：

1. 登录方式改造。
2. 门户表单结构与文案调整。
3. 前后端业务校验规则升级。
4. 成果经历和补充附件的数据结构扩展。
5. 测试、回归与上线准备。

其中：

- 主题 1 和主题 2 可以并行设计，但建议先完成接口方案确认。
- 主题 3 必须在主题 2、主题 4 的结构改动基本稳定后统一落地。
- 主题 5 贯穿全程，但应在每阶段完成后执行小回归。

## 6. 开发计划

## 6.1 第一阶段：需求固化与结构盘点

目标：把本次会议单转成可开发的结构化需求，避免开发过程中反复返工。

任务：

1. 明确“身份证拼接图”是新增单独附件字段，还是复用现有照片类附件。
2. 明确“其他支撑材料”是单文件字段还是单文件列表字段。
3. 取得“个人陈述 Subject QA”原始内容。
4. 明确验证码登录是“邮箱 + 验证码”还是“账号 + 邮箱验证码”。
5. 明确成果经历中“描述”字段是否复用现有 `responsibility_text`，还是新建专用字段。

输出：

- 冻结后的字段清单。
- 冻结后的校验清单。
- 明确需要改数据库结构的项目。

状态建议：优先完成。

## 6.2 第二阶段：登录方式改造

目标：支持“账号密码登录 / 邮箱验证码登录”双模式。

任务：

1. 改造 [frontend/src/views/portal/StudentPortalAuthView.vue](frontend/src/views/portal/StudentPortalAuthView.vue)，新增登录模式切换。
2. 复用或扩展现有邮箱验证码发送能力，区分“注册验证码”和“登录验证码”。
3. 在后端新增或扩展验证码登录接口。
4. 增加验证码有效期、冷却时间、错误次数等保护逻辑。
5. 保持现有账号密码登录可继续使用，避免影响已注册用户。

验收标准：

1. 用户可选择两种登录方式。
2. 验证码登录成功后正常签发门户 token。
3. 错误验证码、过期验证码、被停用账号能得到正确提示。

## 6.3 第三阶段：门户表单结构调整

目标：完成会议单中各章节的字段、布局、文案和默认项调整。

任务：

1. 调整报名信息章节：两个志愿固定展示，文案改为“第一志愿”“第二志愿”，并标记必填/选填。
2. 调整“了解项目方式”的位置。
3. 调整教育经历默认生成逻辑和最多最少条数。
4. 调整高中阶段字段显隐。
5. 调整实践经历默认空白项与最多条数。
6. 调整英语能力章节标题、默认项、选项集，移除 CET-4。
7. 调整家庭成员章节校验口径。
8. 调整成果经历章节标题、类型和动态字段展示。

当前进度补记：第 3、4 项已完成，其余待续。

当前进度补记：第 1、2 项已完成，其余待续。
9. 调整个人陈述和附件区文案。

验收标准：

1. 各章节字段布局符合会议单要求。
2. 默认项生成符合预期。
3. 不同类型切换时显隐字段正确。

## 6.4 第四阶段：后端数据结构与校验升级

目标：让前端新规则在后端也被强制约束，避免“前端校验通过、后端数据脏写”。

任务：

1. 更新 [backend/app/schemas/portal.py](backend/app/schemas/portal.py) 中教育经历、实践经历、英语能力、成果经历、个人陈述等模型。
2. 为教育经历新增跨记录联动校验。
3. 为实践经历新增条件必填校验。
4. 为英语能力新增“附件必填”校验。
5. 为家庭成员新增“父母至少一方完整”校验。
6. 为成果经历扩展论文/获奖两套字段。
7. 为个人简历和其他支撑材料补齐字段及校验。

验收标准：

1. 非法组合数据无法保存草稿或提交。
2. 合法数据可以完整保存并回显。
3. 旧数据回显不会导致接口直接报错。

## 6.5 第五阶段：数据库与持久化改造

目标：将新增字段真正落到 PostgreSQL 物理结构中，不停留在前端或临时 JSON 结构。

任务：

1. 识别成果经历新增字段是否需要改造关系表结构。
2. 识别“其他支撑材料”“身份证拼接图”是否需要新增附件字段或附件表列。
3. 更新 [backend/app/services/postgres_state_store.py](backend/app/services/postgres_state_store.py) 的草稿保存、正式提交、回显装配逻辑。
4. 如涉及数据库结构变化，同步新增迁移 SQL 和生产补丁 SQL。
5. 保持与现有 runtime mirror 的兼容写入。

验收标准：

1. 新字段能持久化。
2. 草稿保存后再次登录能正确回显。
3. 正式提交后后台可见对应字段。

## 6.6 第六阶段：联调、测试与上线准备

目标：在真实门户链路上验证本轮改动不破坏现有报名流程。

任务：

1. 补充后端单元测试，优先覆盖 schema 校验和 portal 提交逻辑。
2. 补充门户端到端烟雾验证，覆盖注册、验证码登录、草稿保存、附件上传、最终提交。
3. 回归后台注册学生管理页与报名信息查看。
4. 如涉及 SQL 变更，整理上线顺序和回滚点。

验收标准：

1. 核心单测通过。
2. 门户主流程可跑通。
3. 后台可正常查看与处理报名数据。

## 7. 建议实施顺序

建议按以下顺序逐步开发：

1. 先做第一阶段，冻结字段和规则。
2. 再做第二阶段，先把登录模式改造独立完成。
3. 再做第三阶段，完成前端各章节结构调整。
4. 然后做第四阶段和第五阶段，统一收口后端校验与落库。
5. 最后做第六阶段联调回归。

原因：

- 登录方式改造与填报表单改造耦合较低，适合先独立完成。
- 教育经历、成果经历属于高变动结构，若前端形态不先稳定，后端 schema 容易重复改。
- 数据库结构改造应建立在字段清单冻结之后，避免重复迁移。

## 8. 风险与注意事项

1. 教育经历的联动规则较强，若只做前端校验，容易被接口绕过，必须前后端双重校验。
2. 验证码登录若直接复用注册验证码，需要注意业务隔离，避免注册验证码串用到登录场景。
3. 成果经历新增字段可能影响现有关系表和历史数据回显，需要做好兼容读取。
4. 个人陈述依赖 Subject QA 原始内容，若业务材料不到位，该章节只能先做结构预留。
5. 附件新增分类若未同步后端白名单，会直接导致上传失败。

## 9. 建议的首个开发任务

建议我们从“第一阶段：需求固化与结构盘点”开始，但如果你希望立刻进入代码开发，我建议优先做下面这个切入点：

1. 先实现“教育经历规则升级”。

原因：

- 这是本轮最复杂、约束最多、最容易引发连锁变更的部分。
- 教育经历一旦稳定，英语能力、家庭成员、实践经历等章节都属于同类规则调整，推进会更顺。
- 它会直接带出前端默认项、后端 schema 校验、保存过滤、草稿回显四类核心实现模式，适合作为本轮开发模板。

## 10. 当前结论

本次 2026-04-27 学生填报需求，不是单纯的页面文案调整，而是一次“门户登录 + 门户填报结构 + 强规则校验 + 局部数据结构扩展”的综合升级。

建议后续按以下节奏逐步推进：

1. 先冻结字段和校验规则。
2. 再先做教育经历或登录方式中的一个主任务。
3. 每完成一个章节，就同步补齐后端校验与最小回归。
