# 解决性能问题计划

## 1. 文档目的

本文用于记录 2026-04-22 对 pydtlms 项目的首轮性能扫描结论，并给出后续整改计划。

状态口径统一如下：

| 状态 | 含义 |
| --- | --- |
| 已完成 | 已确认问题、已有结论，或已完成修复落地 |
| 未完成 | 已识别问题，但尚未完成代码整改或系统验证 |

## 2. 首轮扫描结论

本轮扫描确认，项目当前最主要的性能风险不是单条 SQL，而是以下几类架构性热点：

1. 后端大量写操作仍会落到 `management_service._save()`，触发整份 runtime state 全量写回 PostgreSQL。
2. 一部分列表接口虽然已经完成数据库分页，但仍有一批接口停留在“全量取数 + Python 过滤 + 内存分页”。
3. RuntimeManagementStore 冷启动仍依赖 `load_state()` 整库装载，首个请求成本偏高。
4. 多个管理页首屏会并发拉取列表、统计、选项和参考数据，存在重复请求与大 payload。
5. 部分详情接口、邮件发送链路和 JSONB 查询模式仍有进一步优化空间。

### 3.2 当前高风险热点

| 风险级别 | 问题 | 现象 |
| --- | --- | --- |
| 高 | 高频写接口仍走全量 `_save()` | 数据变更会触发整份 runtime state 重写，数据库写入、JSON 序列化和事务耗时被整体放大。 |
| 高 | 剩余列表仍为内存分页 | 数据量增长后会出现全量读取、全量反序列化和 Python 切片开销。 |
| 高 | 冷启动整库装载 | 首个请求可能被 `load_state()` 及全量 payload 反序列化拖慢。 |
| 中 | 前端首屏与翻页重复拉取 stats/options/reference | 放大前后端请求量，导致“翻页像刷新整页”。 |
| 中 | 详情接口多次串行查询 | 单请求内部存在多次数据库往返，尾延迟明显。 |
| 中 | SMTP 同步发送仍在请求链路 | 外部网络抖动时容易把接口尾延迟拖高。 |

## 7. 当前结论

当前项目的性能改造应继续坚持两条主线：

1. 写路径去全量保存。
2. 读路径去内存分页与全量装载。

在这两条主线完成前，单纯调大超时、增加机器资源或局部调 SQL，都只能缓解症状，不能解决根因。
