# 20260514学生申请全流程开发计划

## 1. 背景

本计划对应客户于 2026-05-14 提供的学生申请全流程新口径，覆盖以下六个业务环节：

1. 在线申请
2. 资料审核
3. 背景评估
4. 中心考核
5. 结果公布
6. 预录取

本轮对话已先完成第一项可见成果：

- 在学生门户首页增加“当前所在环节以及所有环节状态”的方块加箭头展示。

后续工作按照本计划逐步推进，由业务方确认后按阶段实施。

## 2. 目标

1. 将学生门户、注册学生管理、审批中心、研究中心负责人处理、结果公布、预录取确认整合为一条可追踪的统一业务流程。
2. 保证“申请流转状态”和“报名状态”在各页面展示口径一致。
3. 将角色处理权限明确到：书院管理员、研究中心负责人、导师。
4. 将邮件通知、门户展示、审批动作、结果发布联动起来。
5. 尽量复用现有 workflow 与门户报名基础设施，避免平行造第二套状态机。

## 3. 当前现状判断

### 3.1 已有基础

1. 门户在线申请、草稿保存、提交申请已存在。
2. 注册学生列表已能展示报名状态与部分申请流转状态。
3. 后端已有 recruitment_application 的 workflow 托管基础设施。
4. 审批中心、任务详情、动作执行链路已经存在。
5. 门户首页已具备章节完成度展示，本轮已补充环节状态展示。

### 3.2 当前缺口

1. 现有 recruitment workflow 节点定义仍是“资格审核/评分准备/面试安排/录取决策/录取确认”的旧口径，与新业务口径不一致。
2. 缺少“背景评估需 2 位书院管理员通过/不通过”的会签式处理规则。
3. 缺少“中心考核第一志愿/第二志愿”的显式流程状态和权限控制。
4. 缺少“结果公布”由导师提交、书院审核、门户发布的完整闭环。
5. 缺少“预录取接受”倒计时确认、声明弹框、通知书下载与邮件通知链路。
6. 门户首页当前环节条已可展示，但其状态映射仍是兼容层，后续需与正式新流程完全对齐。

## 4. 状态口径冻结建议

建议先冻结以下两个字段的口径，再进入开发：

### 4.1 申请流转状态

建议使用以下中文状态作为统一展示值：

1. 未提交
2. 待审核
3. 驳回重填
4. 待背景评估
5. 待中心考核
6. 结果公布
7. 报名终止
8. 预录取

补充建议：

- 若业务上需要区分第一志愿考核、第二志愿考核，内部可继续保留更细状态值，但前端统一展示口径需冻结。

### 4.2 报名状态

建议冻结为以下展示值：

1. 未提交报名
2. 已填写报名
3. 驳回重填
4. 第一志愿考核
5. 第二志愿考核
6. 已结束报名流程

## 5. 分阶段开发计划

## 5.1 阶段一：门户首页流程可视化

目标：让学生在门户首页看到当前所处环节和所有环节状态。

任务：

1. 后端门户 `/portal/me` 返回流程摘要。
2. 门户首页增加方块加箭头的阶段状态条。
3. 同步显示当前环节、申请流转状态、报名状态、结果公布摘要。
4. 兼容未提交、驳回重填、终止等状态。

状态：本轮已完成首版。

后续补强：

1. 当正式新 workflow 落地后，替换当前兼容映射逻辑。
2. 增加结果公布与预录取的更精细展示文案。

## 5.2 阶段二：流程状态机重构

目标：将现有 recruitment_application workflow 改为符合新业务口径的节点定义。

任务：

1. 重构 workflow 定义，将旧节点替换为：
   - 在线申请
   - 资料审核
   - 背景评估
   - 中心考核
   - 结果公布
   - 预录取
2. 明确每个节点的处理角色、可执行动作、状态回写字段。
3. 明确驳回、终止、通过、转下一志愿等分支。
4. 校验 workflow task、历史记录、business_key 与业务对象状态联动。

验收标准：

1. 托管状态只能由 workflow 动作推进。
2. 审批中心、学生管理、门户首页显示一致。
3. 同一业务对象在 runtime/history 中状态一致。

## 5.3 阶段三：资料审核节点

目标：完成书院管理员对资料审核的处理。

任务：

1. 将“待审核”指派给书院管理员角色。
2. 任意书院管理员可执行“审核通过/驳回重填”。
3. 驳回重填后回到在线申请。
4. 驳回后门户可重新编辑并重新提交。
5. 更新邮件通知与操作日志。

验收标准：

1. 待审核数据只允许有权限角色处理。
2. 审核通过进入“待背景评估”。
3. 驳回重填后门户首页和申请页都能正确回显“驳回重填”。

## 5.4 阶段四：背景评估节点

目标：完成 2 位书院管理员会签式背景评估。

任务：

1. 新增背景评估记录结构，支持多位管理员独立提交结果。
2. 实现“两位通过才进入待中心考核”的规则。
3. 实现“两位不通过则报名终止”的规则。
4. 处理中邮件通知：
   - 通过：通知进入中心考核
   - 不通过：通知报名终止
5. 审批详情页展示评估明细与累计结果。

验收标准：

1. 单人通过不自动流转。
2. 达到 2 位通过后才进入中心考核。
3. 达到终止条件后流程结束并通知学生。

## 5.5 阶段五：中心考核节点

目标：支持第一志愿与第二志愿考核分支。

任务：

1. 为中心考核定义处理角色：第一志愿研究中心负责人或书院管理员。
2. 第一志愿不合格且存在第二志愿时，自动流转到第二志愿考核。
3. 第二志愿处理角色为第二志愿研究中心负责人或书院管理员。
4. 在报名状态中区分：
   - 第一志愿考核
   - 第二志愿考核
5. 处理邮件通知：
   - 第一志愿不合格但有第二志愿：通知待第二志愿考核
   - 考核通过：通知已通过考核，等待结果公布
   - 最终不合格：通知报名结束

验收标准：

1. 单志愿、双志愿分支都能闭环。
2. 第一志愿与第二志愿的处理权限、记录、通知准确。
3. 报名状态切换与业务规则一致。

## 5.6 阶段六：结果公布节点

目标：让“通过/不合格”的最终结果经过导师提交与书院审核后再对学生发布。

任务：

1. 明确导师在系统中的入口：从学生报名详情提交结果公布申请。
2. 新增书院管理员审核发布动作。
3. 门户首页“结果公布”字段只在发布完成后更新。
4. 将中心考核通过结果改为“待结果公布审核”，而不是直接对学生可见。
5. 增加结果发布日志与邮件通知。

验收标准：

1. 中心考核通过后，门户不能提前看到最终结果。
2. 只有结果公布审核完成后，学生端才显示“第一志愿考核通过/第二志愿考核通过/不合格”。
3. 结果内容与考核分支一致。

## 5.7 阶段七：预录取节点

目标：完成学生接受预录取、下载预录取通知书、邮件通知的闭环。

任务：

1. 增加预录取声明弹框。
2. 支持 10 秒倒计时后点击“接受预录取”。
3. 接受后写回状态并生成可下载的预录取通知书。
4. 发送邮件通知。
5. 门户首页与门户申请页展示“已接受预录取”结果。

前置待确认：

1. 预录取声明文本。
2. 通知书模板来源。
3. 是否允许重复下载。

验收标准：

1. 倒计时前不能接受。
2. 接受后状态不可逆或需明确业务规则。
3. 通知书可下载，邮件发送成功。

## 5.8 阶段八：联调、回归与上线准备

目标：完成端到端验证并形成上线清单。

任务：

1. 补齐后端单元测试与 API 测试。
2. 补齐前端关键交互验证。
3. 增加模拟脚本覆盖主要分支：
   - 未提交
   - 驳回重填
   - 背景评估通过
   - 背景评估不通过
   - 第一志愿通过
   - 第一志愿不通过转第二志愿
   - 第二志愿通过
   - 第二志愿不通过
   - 结果公布发布
   - 预录取接受
4. 输出 SQL、发布清单、回滚方案。

## 6. 建议实施顺序

建议按以下顺序推进：

1. 阶段二：流程状态机重构
2. 阶段三：资料审核
3. 阶段四：背景评估
4. 阶段五：中心考核
5. 阶段六：结果公布
6. 阶段七：预录取
7. 阶段八：联调与上线

原因：

- 如果状态机不先重构，后面的门户展示、审批动作、结果公布都会重复返工。
- “结果公布”和“预录取”都依赖前面节点的真实流转结果。

## 7. 需要业务先确认的事项

1. “书院管理员”对应现有哪个角色编码，是否新增专门角色。
2. “研究中心负责人”是否直接复用团队负责人账号。
3. 背景评估是“恰好两位管理员”还是“至少两位管理员”通过。
4. 背景评估不通过的判定规则是否也是“两位不通过即终止”。
5. 导师提交结果公布时，是否只能提交与中心考核结论一致的结果。
6. 预录取声明文本、通知书模板、下载格式。
7. 第二志愿考核通知邮件文案、结果公布文案。

## 8. 技术风险提示

1. 当前 recruitment workflow 定义与新业务口径差异较大，不能只改文案，需要实际重构节点与动作。
2. “背景评估双人通过”不属于当前简单单任务流转，可能需要增加会签/累计判断逻辑。
3. “结果公布”与“预录取”新增后，门户首页、学生管理、审批中心、邮件通知要统一改动。
4. 若数据库结构要持久化背景评估明细、结果公布记录、预录取接受记录，则需要新增表或新增字段。

2. 门户 `/portal/me` 返回流程状态摘要字段，供首页消费。
3. 形成本开发计划文档，作为后续逐步推进的基线。
